# 0단계 스펙 — 프로젝트 뼈대

> **전제**: 먼저 마스터 스펙(`session2_master_spec.md`)의 불변 규칙·폴더 구조·작업 방식을 읽고 따른다. 충돌 시 **마스터 스펙이 우선**한다.
> **범위**: 이 문서는 **0단계만** 구현한다. 1단계 이후 내용(실제 데이터 다운로드 콘텐츠, 모델, 라벨링, 증강 등)은 **만들지 않는다**.
>
> **현행 기준 안내(멀티세션 마이그레이션 후)**: 이 문서는 참고용/이력이다. 범용 규칙의 최신·단일 출처는 레포 루트 `CONVENTIONS.md` 이며, 실제 폴더 경로·Colab 배지는 `session2/` 기준([`session2/README.md`](../session2/README.md))을 따른다. 본문에 나오는 `notebooks/00_setup.ipynb` 등 플랫 경로는 `session2/notebooks/00_setup.ipynb` 로 읽는다.

---

## 1. 이 단계의 목적

프로젝트 골격을 세우고 **"빈 셋업 노트북이 로컬과 코랩 양쪽에서 에러 없이 실행된다"**를 보장한다. 이후 모든 단계(1~5)가 이 골격 위에 쌓이므로, 여기서 경로·환경·버전 문제를 전부 잡아둔다.

---

## 2. 만들 산출물

1. **폴더 구조** — 마스터 스펙 4장의 구조대로 생성 (`data/`, `weights/`, `outputs/`, `notebooks/`).
2. **`requirements.txt`** — 베이스라인 의존성만, 버전 고정.
3. **`README.md`** — 배포·실행 안내(특히 학생용 준비 절차).
4. **`notebooks/00_setup.ipynb`** — 환경 확인 + 폴더 준비 + 데이터 다운로드 자리(골격).

---

## 3. `00_setup.ipynb` 구성

각 셀은 **독립 실행 가능**해야 하고(마스터 규칙 4), 맨 위에 **헤더 3줄**(규칙 6)을 둔다.

**헤더 (마크다운)**
- 이 노트북에서 배우는 것 / 입력 / 출력 — 3줄. (이 노트북은 "개발환경이 준비됐는지 확인하는 곳"이라는 취지)

**셀 1 — 환경 감지 · 프로젝트 루트 확보**
- 코랩/로컬을 감지한다. 감지는 표준 방식 사용:
  ```python
  try:
      import google.colab
      IN_COLAB = True
  except ImportError:
      IN_COLAB = False
  ```
- **로컬**: `requirements.txt`를 마커로 삼아 현재 경로에서 상위로 올라가며 탐색해 **프로젝트 루트**를 찾는다(실행 위치와 무관하게 동작하도록). 그 위치로 `chdir`.
- **코랩**: 아래 4장의 "프로젝트 확보" 규칙대로 프로젝트를 작업공간으로 가져온 뒤 그 위치로 `chdir`.
- 이후 모든 경로는 **프로젝트 루트 기준 상대경로**로 쓴다(규칙 1). `PROJECT_ROOT` 변수만 잡아두고, 코드에서는 `data/`, `outputs/` 같은 상대경로를 사용한다.

**셀 2 — 의존성 설치**
- `requirements.txt`로 설치한다. **코랩에 이미 설치된 패키지는 재설치하지 않도록** 한다(특히 torch는 코랩 사전설치 버전과 충돌·대용량 재설치가 일어나지 않게 — 4장 참고).

**셀 3 — 폴더 생성**
- `data/`, `weights/`, `outputs/`가 없으면 생성한다(있으면 그대로 둠). 멱등하게.

**셀 4 — 데이터 다운로드 자리 (골격만)**
- 실제 다운로드 콘텐츠는 **1단계에서 채운다.** 지금은 빈 함수(또는 명확한 주석)와 "여기서 1단계에 샘플 데이터가 받아진다"는 안내만 둔다. **실제 데이터를 받지 않는다.**

**셀 5 — 검증 셀 (통과 신호)**
- 한 번에 다음을 사람이 보기 좋게 출력한다:
  - 실행 환경(코랩/로컬)
  - `PROJECT_ROOT` 경로
  - GPU 사용 가능 여부와 장치명(없으면 "GPU 없음"으로 정상 출력)
  - `data/`, `weights/`, `outputs/` 존재 여부
- 이 셀의 출력만 보면 0단계가 통과인지 한눈에 판단되게 한다.

## 4. 환경 분기 규칙 (반드시 한 곳에서)

로컬/코랩 차이는 **셀 1의 한 곳에서만** 처리하고, 그 외 코드는 양쪽에서 동일하게 동작한다(규칙 7).

- **코랩의 "프로젝트 확보" 방식은 한 함수(예: `acquire_project()`)에 격리**한다. 배포 방식이 바뀌어도 이 함수와 README의 배지/링크만 교체하면 되게 한다.
- **기본 = 공개 GitHub 레포 클론.** 레포: `https://github.com/BSLJunhyeonJeon/AI_COP` (Public, 기본 브랜치 `main`).
  - "Open in Colab" 배지가 GitHub의 `00_setup.ipynb`를 직접 열고, **첫 셀에서 레포를 `/content`로 클론**해 나머지 파일을 확보한다. 공개 레포라 학생은 **토큰·계정·인증이 전혀 필요 없다**(익명 clone).
  - 클론 위치: **`/content/AI_COP`**. 이곳이 코랩에서의 `PROJECT_ROOT`가 되며, `chdir` 후 모든 경로는 상대경로.
  - **멱등 처리**: 이미 `/content/AI_COP`가 있으면(재실행) 다시 클론하지 말고 건너뛰거나 `git pull` 후 `chdir`. clone 실패로 셀이 죽지 않게 방어한다.
- **로컬**: 레포를 클론한 폴더가 작업 루트다. `requirements.txt`를 마커로 상위 탐색해 `PROJECT_ROOT`를 잡고 `chdir`(셀 1 로컬 분기와 동일).
- **참고(교체 가능)**: 드라이브 배포로 바꾸려면 `acquire_project()`(드라이브 마운트 + 폴더 복사)와 배지/README 절차만 교체하면 된다.

**Colab 배지 URL** (README와 `00_setup.ipynb` 상단에 사용)
```
https://colab.research.google.com/github/BSLJunhyeonJeon/AI_COP/blob/main/notebooks/00_setup.ipynb
```

> 어느 방식이든 **셀 5 검증 셀은 동일**하게 동작해야 한다.

## 5. `requirements.txt`

- **베이스라인만** 넣는다(현재 셋업에 실제로 필요한 것). 사실상 GPU 확인용 `torch` 정도이며, 다운로드 유틸 등은 **1단계에서 필요해질 때 추가**한다.
- **버전은 코랩에서 실제로 확인된 값으로 고정**한다. 임의의 버전을 추측해 적지 말 것.
- **torch는 코랩에 사전설치된 버전과 정확히 일치하도록 고정**해, 코랩에서 재설치가 트리거되지 않게 한다. 로컬에서는 이 버전을 설치해 코랩과 환경을 맞춘다.

## 6. `README.md`

다음을 간단히(초보자 기준) 담는다:
- 프로젝트 한 줄 소개(2회차 학습 자료).
- **상단에 "Open in Colab" 배지** — 클릭하면 `00_setup.ipynb`가 코랩에서 열린다. 배지 대상은 4장의 URL.
- **학생 준비 절차** — (1) 배지 클릭 → 코랩 열림, (2) 위에서부터 셀 ▶ 순서대로 실행. 끝. (공개 레포라 로그인·토큰 불필요, 드라이브 마운트 없음)
- **로컬 실행법** — 짧게: 레포 클론 → 환경 만들고 `requirements.txt` 설치 → 노트북 실행.

- [ ] **로컬**에서도 동일 실행 → 에러 없음(GPU 없으면 "GPU 없음"으로 정상 출력).
- [ ] `PROJECT_ROOT`가 양쪽에서 올바르게 잡히고, 상대경로로 폴더에 접근됨.
